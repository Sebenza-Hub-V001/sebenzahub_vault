# Plan: Recurring AI Monitoring Loops + Multi-Agent Screening Workflow

## Context

Sebenza Hub has a mature AI Skills Registry, BullMQ job queues, screening workflows, and compliance monitoring — but these systems operate in isolation. This plan adds two architectural capabilities:

1. **Recurring AI Monitoring Loops** — Scheduled, configurable monitors that leverage AI Skills to detect stale jobs, audit compliance, refresh candidate matches, and check WhatsApp bot health.
2. **Multi-Agent Screening Workflow** — Replaces the monolithic single-AI-call screening with 4 composable, auditable agents (CV Parser, Competency, Compliance, Culture-Fit) coordinated by an orchestrator.

Both features build on existing infrastructure (BullMQ, AI Skills Registry, screening tables, compliance routes) rather than creating parallel systems.

## Feature 1: Recurring AI Monitoring Loops

### 1.1 Database Schema (3 new tables in `shared/schema.ts`)

**`ai_recurring_monitors`** — Configuration for each monitor loop

- `id`, `monitorKey` (unique), `name`, `description`, `category` (stale_jobs | compliance_audit | candidate_match_refresh | whatsapp_health)
- `skillId` FK -> `aiSkillPrompts.id` (links to AI Skills Registry)
- `cronPattern` (e.g., "0 2 * * *"), `intervalMinutes` (UI-friendly alternative)
- `isEnabled`, `config` (JSONB — monitor-specific thresholds), `lastRunAt`, `lastRunStatus`, `nextRunAt`
- `consecutiveFailures`, `maxRetries`, `timeoutMs`, `createdBy`, timestamps

**`ai_recurring_monitor_runs`** — Execution log per run

- `id`, `monitorId` FK, `status` (running|success|failed|timeout|partial)
- `startedAt`, `completedAt`, `durationMs`, `itemsProcessed`, `itemsFlagged`
- `summary` (JSONB), `errorMessage`, `aiTokensUsed`, `aiCostUsd`, `triggeredBy`

**`ai_recurring_monitor_alerts`** — Alerts generated by runs

- `id`, `monitorId` FK, `runId` FK, `severity` (critical|high|medium|low|info)
- `title`, `description`, `entityType` (job|consent|certificate|candidate|conversation), `entityId`
- `actionRequired`, `actionTaken`, `status` (open|acknowledged|resolved|dismissed), `metadata` (JSONB)

### 1.2 Backend Architecture

**New BullMQ queue in `/server/queue.ts`:**

- Add `aiRecurringMonitorQueue` following the existing queue pattern (lines 9-18, 61-70)
- Single worker dispatching to handlers by `job.name`
- Repeatable jobs registered at startup from enabled monitors in DB
- Concurrency: 2

**New file: `/server/recurring-monitors.ts`** — Core execution engine:

- `executeMonitorRun(monitorId, triggeredBy)` — orchestrator that loads config, creates run record, dispatches to handler, records results
- 4 handler functions:

|Monitor|What it does|Key data sources|
|---|---|---|
|`runStaleJobDetection`|Scans `jobs` for outdated listings by `updatedAt`/`expiresAt`|`jobs` table, configurable threshold|
|`runComplianceAudit`|Checks POPIA consent expiry, B-BBEE cert renewal, EE report deadlines|`popiaConsents`, `bbbeeScores`, `eeReports` — reuse queries from `admin-compliance-enhanced-routes.ts`|
|`runCandidateMatchRefresh`|Re-runs matching for jobs/candidates updated since last run|`matching.ts` scoring functions|
|`runWhatsAppHealthMonitoring`|Checks bot error rates, stuck conversations, response times|`whatsapp-analytics.ts` `computeAnalytics()`, `whatsappMessages` table|

Each handler optionally uses its linked AI Skill prompt for deeper analysis.

**Sync mechanism:** When admin toggles/updates a monitor, API removes old repeatable job via `queue.removeRepeatableByKey()` and re-adds with new config.

## Feature 2: Multi-Agent Screening Workflow

### 2.1 Agent Architecture

Each agent is a function wrapping `chatJSON` from `/server/ai.ts`, using its own `featureKey` to resolve prompts from AI Skills Registry.

**4 Agents:**

|Agent|featureKey|Input|Output|
|---|---|---|---|
|CV Parser|`screeningAgentCvParser`|Raw CV text|Structured data + completeness score|
|Competency|`screeningAgentCompetency`|Parsed CV + job requirements|Skills/experience match score + knockouts|
|Compliance|`screeningAgentCompliance`|Parsed CV + demographics + org B-BBEE targets|Compliance readiness score + regulatory flags|
|Culture-Fit|`screeningAgentCultureFit`|Parsed CV + team DNA profiles|Fit score + dimension alignments|

**Agent result interface:**

```typescript
interface AgentResult {
  agentKey: string;       // "cv-parser" | "competency" | "compliance" | "culture-fit"
  score: number;          // 0-100
  confidence: number;     // 0-100
  breakdown: Record<string, number>;
  flags: string[];
  reasoning: string;
  metadata: Record<string, any>;
  durationMs: number;
  status: "completed" | "failed" | "skipped";
  error?: string;
}
```

**New file: `/server/screening-agents.ts`** — All 4 agent functions.

### 2.2 Orchestrator Design

**New file: `/server/screening-orchestrator.ts`**

**Single-candidate flow:** `orchestrateScreening(candidate, job, options)`

1. Pre-check: load data, check if culture-fit team profiles exist
2. Dispatch all agents in parallel via `Promise.allSettled`
3. Collect results (failed agents produce error `AgentResult`, don't block others)
4. Merge into `OrchestrationResult`:
    - Combined weighted score (default: CV 10%, Competency 50%, Compliance 20%, Culture-Fit 20%)
    - If culture-fit skipped: CV 10%, Competency 60%, Compliance 30%
    - Knockouts from Competency override score to 0
5. Recommended action: `reject` (<30 or knockout), `shortlist` (>=70 + confidence >=60), `review` (else)

**Fallback:** If Competency agent fails, fall back to existing monolithic `screenCandidate()`.

### 2.3 Database Schema (2 new tables + column additions)

**`screening_agent_results`** — Per-agent results for audit trail

- `id`, `screeningJobId` FK, `screeningCandidateId` FK, `screeningEvaluationId` FK (nullable)
- `agentKey`, `score`, `confidence`, `breakdown` (JSONB), `flags` (JSONB), `reasoning`
- `metadata` (JSONB), `durationMs`, `status`, `errorMessage`, `aiModelUsed`, `aiTokensUsed`

**`screening_orchestration_runs`** — Bulk screening tracking

- `id`, `screeningJobId` FK, `triggeredBy` FK, `mode` (single|bulk)
- `totalCandidates`, `completedCandidates`, `failedCandidates`, `status`
- `agentConfig` (JSONB — which agents, weights), timestamps

**Column additions to `screening_evaluations`:**

- `orchestration_run_id` FK (nullable)
- `evaluation_version` varchar DEFAULT 'v1' ('v1' = legacy, 'v2' = multi-agent)

### 2.4 BullMQ Integration

**New queue: `screening-agents`** in `/server/queue.ts`:

- Job `orchestrate-single`: runs 4 agents for one candidate
- Job `orchestrate-bulk`: iterates candidates, enqueues child jobs
- Concurrency: 5 (= max 20 concurrent AI calls: 5 candidates x 4 agents)

### 2.5 API Route Changes

**Modify existing endpoints in `/server/routes.ts`:**

- `POST /api/screening-jobs/:id/screen-candidate/:candidateId` — use orchestrator (synchronous)
- `POST /api/screening-jobs/:id/bulk-screen` — use queue-based approach, return `runId`

**New endpoints:**

- `GET /api/screening-jobs/:id/screening-progress` — bulk run status + per-candidate completion
- `GET /api/screening-jobs/:id/candidates/:cid/agent-results` — per-agent detail
- `GET /api/screening-jobs/:id/agent-summary` — aggregate stats by agent across all candidates

### 2.6 Frontend Changes (`RecruiterScreening.tsx`)

- **Per-agent results display:** When `evaluationVersion === "v2"`, show accordion/tabs with each agent's score, confidence, flags, reasoning
- **Bulk progress tracker:** Replace synchronous modal with live progress bar polling `/screening-progress`
- **Agent config panel:** Toggle agents, adjust weights (stored in `screeningJobs.weights` JSONB)

### 2.7 Seed Data

**4 new AI Skills** in `SEED_SKILLS`:

- `screening-agent-cv-parser`, `screening-agent-competency`, `screening-agent-compliance`, `screening-agent-culture-fit`

**4 new DEFAULT_MODELS** entries in `/server/ai.ts`.
